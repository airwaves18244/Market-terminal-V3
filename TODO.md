# TODO V3 — план работ с распределением по моделям Claude

Источник требований: `V3/PRD.md`. Нумерация задач `T-<milestone>.<n>`,
ссылки `E*.*` — на разделы PRD.

## Как распределены модели и усилия

| Модель | Роль в проекте | Типичные задачи |
|---|---|---|
| **Fable 5** (`claude-fable-5`) | архитектор/ревьюер | проектные решения, сложные алгоритмы с математикой, security-ревью live-трейдинга, финальное код-ревью мейлстоуна |
| **Opus 4.8** (`claude-opus-4-8`) | ведущий инженер | многокомпонентные фичи через слои (domain+data+storage+app+front), производительность, конкурентность |
| **Sonnet 5** (`claude-sonnet-5`) | основной исполнитель | типовые фичи по готовому контракту, UI-компоненты, IPC-обвязка, тесты, портирование |
| **Haiku 4.5** (`claude-haiku-4-5`) | быстрые правки | моки, DTO/boilerplate, доки, мелкие фиксы, форматирование данных |

Уровни усилий (reasoning effort): **low / medium / high / max**.
Правило: контракт и `(verify)`-фикстуры делает старшая модель на high/max,
массовую реализацию по зафиксированному контракту — младшая на medium.
Каждая задача заканчивается зелёным CI (`fmt`, `clippy -D warnings`, тесты,
svelte-check/vitest) — это входит в Definition of Done и отдельно не пишется.

## Порядок и параллелизм

- Критический путь: **T-1.1 → T-1.3/T-1.4 → M3 → M6**.
- M2 стартует параллельно M1 (mock-first): T-2.1/T-2.4 не ждут сети.
- T-4.1 ждёт только T-1.5/T-1.6 (тиковая история в каталоге).
- T-5.1 (спайк) — как можно раньше внутри M5: от go/no-go зависит объём T-5.2.
- `*.R`-ревью закрывает мейлстоун; найденные дефекты чинит модель-исполнитель
  исходной задачи (тем же effort), Haiku 4.5 подхватывает тривиальные.
